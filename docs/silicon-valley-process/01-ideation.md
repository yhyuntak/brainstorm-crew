# 1. 아이디어 발굴 (Ideation)

> 프로젝트의 시작점: 문제 발견부터 초기 검증까지

---

## 1.1 아이디어의 출발점

실리콘밸리에서 새 프로젝트는 보통 다음 중 하나에서 시작됩니다:

### A. Top-Down (경영진 주도)
```
CEO/VP → Strategic Initiative → Project Proposal
```
- **예시**: "우리는 AI 시장에 진입해야 한다"
- **특징**: 명확한 비즈니스 목표, 충분한 리소스 확보
- **리스크**: 현장 인사이트 부족 가능성

### B. Bottom-Up (엔지니어/PM 주도)
```
문제 발견 → 해결책 프로토타입 → 내부 데모 → 승인
```
- **예시**: Google의 20% 프로젝트, Hackathon 결과물
- **특징**: 실제 문제 기반, 높은 팀 동기부여
- **리스크**: 비즈니스 정당성 입증 필요

### C. Customer-Driven (고객 요청)
```
고객 피드백 → Pain Point 분석 → 솔루션 제안
```
- **예시**: 대규모 고객사의 Feature Request
- **특징**: 명확한 PMF (Product-Market Fit)
- **리스크**: 한 고객에게만 맞춤화될 위험

---

## 1.2 초기 검증 (Pre-Validation)

아이디어가 나오면 바로 프로젝트화하지 않습니다. 먼저:

### 1.2.1 One-Pager 작성 (1-2일)

```markdown
# Project Title

**Problem**: 우리가 해결하려는 문제가 무엇인가?
**Solution**: 어떻게 해결할 것인가? (High-level)
**Impact**: 비즈니스/사용자에게 어떤 영향을 주는가?
**Resources**: 대략 얼마의 시간과 인력이 필요한가?
**Risk**: 주요 리스크는 무엇인가?
```

### 1.2.2 스몰 서클 검증 (2-3일)

- **기술 리더**: "기술적으로 가능한가?"
- **제품 리더**: "사용자가 원하는가?"
- **비즈니스 리더**: "돈이 되는가?"

> **실제 사례**: AWS에서 새로운 서비스 아이디어는 먼저 "Bar Raiser" 2-3명에게 비공식적으로 보여줬습니다. 60% 이상이 여기서 걸러집니다.

### 1.2.3 Spike / Proof of Concept (선택적, 1-2주)

기술적 불확실성이 크면:
- 1-2주 짜리 기술 검증
- "이게 정말 가능한가?" 입증
- 예산 청구 전 리스크 제거

---

## 체크리스트

- [ ] 문제가 명확하게 정의되었는가?
- [ ] One-Pager 작성 완료
- [ ] 3명 이상의 리더에게 피드백 받음
- [ ] 기술적 불확실성이 있다면 PoC 완료
- [ ] Go/No-Go 결정 완료
